Primitivas Coletivas

O trabalho exige pelo menos 2. Usamos 3 na versão otimizada.

bcast

Transmite parâmetros do raiz para todos os processos.

# 2 inteiros para todos os workers
comm.bcast(num_alunos, root=0)
comm.bcast(num_disc, root=0)

Dados trafegados: ~8 bytes

Scatterv

Distribui pedaços de tamanhos diferentes dos arrays.

# 12M floats → pedaços por worker
comm.Scatterv([notas, sendcounts,
   displacements, MPI.DOUBLE],
   array_notas, root=0)

Dados trafegados: ~96 MB total

Gatherv

Coleta resultados parciais de volta ao raiz.

# CRs calculados → raiz
comm.Gatherv(local_cr,
   [array_cr, recvcounts,
    displacements, MPI.DOUBLE],
   root=0)

Dados trafegados: ~800 KB

V1 usa bcast + gather (2 primitivas)  |  V2/V3 usam bcast + Scatterv + gather/Gatherv (3 primitivas)

Por que o Scatterv não escalou?

Lei de Amdahl: o speedup é limitado pela fração serial do programa.

Broadcast (V0/V1)

Geração dados

~100% paralelizável. Cada processo gera e calcula independente. Broadcast só envia 2 inteiros.

~1s

Gargalo serial de 6 segundos. O raiz gera 1,5M floats sozinho. Workers recebem e calculam em milissegundos. A parte serial domina.

Lição: Paralelismo só compensa quando a fração paralelizável é grande e a comunicação é barata. Adicionar processos a uma tarefa com gargalo serial piora o desempenho (overhead de coordenação).

Código

# Cria RDD com índices dos alunos
alunos = np.arange(num_alunos)
rdd = sc.parallelize(alunos, PARTICOES)

# Aplica calc_cr em paralelo
list_cr = rdd.map(calc_cr).collect()

Spark abstrai completamente a distribuição:
sem bcast, sem Scatterv, sem sendcounts.
